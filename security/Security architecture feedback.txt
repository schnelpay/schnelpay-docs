Highest Priority Findings

1. Hybrid Verification Logic Is the Core Security Problem

Your current logic:

if (ecdsaValid || dilithiumValid)
    token.valid = true

Effectively means:

“The attacker only needs to break one scheme.”

That defeats the purpose of hybrid PQ authentication.

In a true hybrid model:

* BOTH signatures should verify
* BOTH signatures should be required
* Failure of either should reject the token

Otherwise the system security collapses to the weakest surviving primitive.

Right now:

* compromise JWT_SECRET → forge “quantum-safe” token
* break Dilithium implementation → forge token
* exploit legacy JWT → bypass PQ guarantees entirely

Severity: Critical

Recommended remediation:

valid = ecdsaValid && dilithiumValid

Then enforce:

* signatures must both exist
* exact algorithm suite expected
* no partial verification acceptance

You can still log which failed internally for diagnostics.

⸻

2. Legacy JWT Fallback Is a Downgrade Vector

This is the second critical issue.

Current behavior:

* PQ verification fails
* system silently attempts legacy JWT verification
* authenticateQuantumSafe accepts it

An attacker’s perspective:

“I do not need to attack Dilithium at all.”

This collapses your security model back to classical JWT.

Routes currently exposed:

* wallet
* payment
* transaction
* portfolio
* accounting
* trust

Those are financially sensitive routes.

Recommendation:

* remove fallback entirely for protected financial APIs
* migrate all protected routes to requireQuantumSafe
* reserve legacy JWT only for:
    * migration endpoints
    * temporary compatibility endpoints
    * explicit versioned auth routes

Ideal model:

/api/v1/legacy/*
/api/v2/quantum/*

Then deprecate v1 aggressively.

Severity: Critical

⸻

3. “ECDSA” Naming Is Dangerous

This is not cosmetic.

You currently describe:

ecdsa = HMAC-SHA256

That is materially different cryptography.

HMAC:

* symmetric
* shared-secret MAC
* server-verifiable only

ECDSA:

* asymmetric
* public/private keypair
* non-repudiable signatures

An external reviewer, partner, or future engineer could make incorrect security assumptions.

Recommendation:
Rename immediately:

* ecdsa → hmac
* signWithECDSA → signWithHMAC
* algorithm: ecdsa → algorithm: hmac-sha256

Severity: High

⸻

4. Single Secret Used Across Multiple Security Domains

Using JWT_SECRET for:

* legacy JWT signing
* hybrid HMAC authentication

creates coupling between:

* migration compatibility
* primary auth integrity

If JWT_SECRET leaks:

* attacker can forge legacy JWT
* attacker can forge hybrid classical half
* because of your OR logic, attacker wins immediately

Even after fixing AND verification, this is still undesirable.

Recommendation:
Separate secrets entirely:

JWT_LEGACY_SECRET=
HYBRID_HMAC_SECRET=

Even better:

* move classical hybrid side to Ed25519 instead of HMAC

That would give:

* asymmetric hybrid
* cleaner architecture
* easier external verification
* stronger future audit posture

Severity: High

Additional Important Findings

5. Revocation Fail-Open

You already spotted this correctly.

If Redis fails:

* revoked tokens may become valid again

For fintech/payment systems, this is dangerous.

Recommended behavior:

* fail closed for high-risk operations
* or use degraded auth mode

Example:

* read-only APIs may tolerate fail-open
* withdrawal/payment/admin should fail-closed

Suggested architecture:

* local short-lived revocation cache
* Redis cluster redundancy
* revocation epoch/versioning

Severity: High

⸻

6. JSON Re-serialization Risks

This part matters more than many engineers realize.

If signing uses:

JSON.stringify(payload)

you are depending on:

* deterministic property order
* deterministic serialization
* identical runtime behavior

This is fragile long-term.

Use canonical serialization:

* RFC 8785 JSON Canonicalization Scheme (JCS)
* CBOR canonical encoding
* Protocol Buffers canonical encoding

Right now it may work consistently in Node.js, but cross-language verification later could fail.

Severity: Medium-High

⸻

7. Token Self-Describing Algorithm Fields

You asked an excellent question here.

Never trust token-declared algorithms.

If token contains:

{
  "algorithm": "ecdsa"
}

the verifier must NOT dynamically choose logic based on attacker-controlled metadata.

Verifier should enforce:

EXPECTED_ALGORITHM_SUITE = HYBRID_V1

Hardcoded server-side.

Do not permit:

* algorithm negotiation
* optional algorithms
* runtime fallback

Severity: High

⸻

8. TOTP Plaintext Storage

This is acceptable for many systems if:

* TTL short
* Redis isolated
* transport encrypted
* no persistence

But for a platform branding itself around advanced security, hashing OTPs is preferable.

Recommended:

stored = SHA256(code + nonce)

Not critical, but improves posture.

Severity: Low-Medium

⸻

9. Password Policy Enforcement Gap

You likely have:

* validator exists
* not called during registration path

This is common.

Recommendation:
Enforce centrally:

validatePasswordStrength(password)

inside:

* register
* resetPassword
* admin password reset
* invitation flows

Severity: Medium
