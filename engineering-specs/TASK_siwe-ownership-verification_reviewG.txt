## 1. Smart Wallet Verification Path (ERC-1271 & ERC-6492)
Your approach using viem is completely sound, but the exact execution details have a few hidden traps that can cause silent production failures (p. 2):

* Action vs. Utility Gotcha: In viem, the standard utility function verifyMessage or verifySiweMessage imported directly from viem/utils or viem/siwe only supports EOAs (it relies strictly on local ecrecover). To utilize the ERC-1271 / ERC-6492 path, you must use the Public Client Action (publicClient.verifyMessage or publicClient.verifySiweMessage). Your specification mentions await publicClientFor(chainId).verifySiweMessage(...), which is correct (p. 3). Keep this distinction sharp during coding. [1, 2, 3, 4] 
* The Chain ID / RPC Mismatch Risk: Coinbase Smart Wallets generate signatures that encode state specifically bound to the network they live on (predominantly Base, chainId 8453) (p. 2).
* The Trap: If a user connects via Base, but the frontend generates a SIWE message with chainId: 1 (Ethereum Mainnet), or if your backend utilizes the Mainnet Public Client to call verifySiweMessage on a Base smart wallet signature, the validation will fail completely or default to a failed eth_call (p. 2).
   * The Fix: Your publicClientFor(chainId) abstractor must reliably map the chainId parsed dynamically out of the message to its matching RPC endpoint (RPC_URL_BASE vs RPC_URL_MAINNET) (pp. 2-3). Never default to a single global client for verification.
* Deployless Universal Signature Verifier: Under the hood, viem handles ERC-6492 by executing an eth_call to a deployless multicall / verifier contract. This requires your RPC endpoints to have robust uptime and standard JSON-RPC capabilities. Make sure your Infura/RPC configuration doesn't strictly block custom eth_call payloads simulating state overrides. [5, 6]

## 2. The Nonce Lifecycle & Replay Protection
Your plan utilizes an optimal security pattern, but you need to account for specific race conditions and multi-region infrastructure layouts (pp. 1, 3):
 
* Atomic Verification via GETDEL: Using Redis GETDEL is the definitive gold standard for preventing replay attacks (p. 3). It ensures that the moment a nonce is read, it is destroyed instantly in a single atomic transaction (p. 3). This eliminates time-of-check to time-of-use (TOCTOU) race conditions where a user could rapidly double-submit the exact same payload before a standalone DEL command executes.
* User Binding Matrix: Your Redis key structure siwe:nonce:{userId} perfectly binds the lifecycle to the authenticated session (p. 3). This prevents a malicious attacker from requesting a valid nonce on their own account and submitting it inside a hijacked verification payload for a target user's wallet address.
* Clock Skew Grace Periods: You mention checking if issuedAt is implausibly old (p. 3). Ensure your backend applies a strict maximum drift threshold (e.g., ±60 seconds). Because client-side device clocks can occasionally lag behind atomic server time, failing to tolerate minor clock skew can result in false-positive rejections for legitimate users.
