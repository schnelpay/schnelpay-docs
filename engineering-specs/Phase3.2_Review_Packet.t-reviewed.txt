This is a strong design pattern. The core security model—approve → immutable signed snapshot → enforced delay → re-verify before broadcast—is a standard and defensible mitigation against TOCTOU attacks in custodial systems.

My review focuses on what could still go wrong.

1. Threat Model Gaps

A. Destination Is Potentially Incomplete (HIGH)

You asked the right question.

The biggest thing I would challenge is:

destination

is not universally sufficient across blockchains.

Examples:

XRP

Requires:

address
destination_tag

Changing destination tag can redirect funds.

Stellar

Requires:

address
memo

Memo change can redirect funds.

Cosmos-family chains

May require:

address
memo

Exchange deposit addresses

Often:

address + memo/tag

is the true destination.

Recommendation

The signed payload should contain:

destination_canonical

where canonical means:

address
memo/tag
network-specific routing fields

serialized into a normalized string.

Example:

{
  "address":"...",
  "memo":"12345"
}

or

address|memo

after canonicalization.

If SchnelPay ever supports memo/tag chains, current signing is insufficient.

⸻

B. Blockchain Name Ambiguity (HIGH)

You specifically asked about:

chain-id vs blockchain-name ambiguity

I agree this deserves attention.

Example:

USDC

exists on:

* Ethereum
* Base
* Arbitrum
* Polygon
* Solana

If:

blockchain = "ethereum"

is stored today but later internal routing uses:

network_id
chain_id
asset_network

you could create ambiguity.

Recommendation

Sign:

chain_id

or

network_identifier

rather than a human-readable label.

Example:

ethereum-mainnet
base-mainnet
polygon-mainnet

or numeric chain IDs where applicable.

⸻

C. Asset Identity Not Mentioned (Potential High)

Your payload includes:

amount
blockchain
destination

but I do not see:

asset
currency
token

listed.

Question:

Can a withdrawal row ever change from:

USDC

to

USDT

without changing amount?

If yes:

This must be signed.

A withdrawal approval should bind:

asset
amount
destination
network

as a single unit.

If asset already exists elsewhere and is immutable, then this is informational only.

3. Concurrency Review

Double Approval

You appear protected by:

FOR UPDATE

plus

UNIQUE(tx_id)

This is the correct combination.

No obvious issue.

⸻

Approval vs Cancel Race (MEDIUM)

I would explicitly test:

Thread A:

approveWithdrawal()

Thread B:

cancelWithdrawal()

at nearly the same instant.

Questions:

1. Does cancel acquire same lock?
2. Can cancellation occur after snapshot creation but before status update?
3. Can verify see inconsistent state?

Without code I cannot prove safety.

I recommend adding an explicit race test.

⸻

Verify vs Manual Modification Race

Suppose:

verifyBeforeBroadcast()

passes

and then:

broadcast()

occurs 2 seconds later.

Can anything modify state between those two actions?

Ideally:

verify + transition to broadcasting

should occur inside one transactional ownership step.

Otherwise a second TOCTOU window exists.

Not necessarily exploitable today, but worth documenting.

5. Security Hold Recovery

The design intentionally dead-ends.

Good.

However I’d add:

Required Admin Workflow

Document:

1. investigate
2. create replacement withdrawal
3. never edit original
4. original remains immutable

Avoid:

release from security_hold

without generating a new approval snapshot.

Otherwise operators may bypass the security boundary.

⸻

Audit Trail

I would want:

who
when
why

for security_hold disposition actions.

Even if implemented later.

6. DB Immutability Review

Your triggers are strong against application bugs.

But they are not absolute.

PostgreSQL Superuser

Can:

ALTER TABLE
DROP TRIGGER
DISABLE TRIGGER
UPDATE ...

Migration Role

May also bypass controls depending on privileges.

Therefore:

The trigger provides:

application-level immutability

not

cryptographic immutability

I would simply document:

Ledger immutability assumes trusted DB administrators.

That’s industry standard.

7. Secret Rotation (MEDIUM)

This is the biggest operational weakness.

Current rule:

never rotate

is dangerous long-term.

Better Design

Store:

key_id
signature
payload

Example:

key_id = 3

Verification:

lookup key 3
verify

Now:

* old approvals continue verifying
* new approvals use new key

This is how KMS-backed signing systems typically evolve.

I would strongly recommend planning for this before production.

⸻

8. Signed Field Maintenance (MEDIUM)

I agree with your own concern.

Current model:

add field
remember to sign field
remember to verify field

Human-error prone.

Better Pattern

Define one canonical object:

buildApprovalSnapshot(tx)

used by BOTH:

sign()
verify()

and ideally generated from a single schema definition.

Then adding a protected field becomes a one-location change.

Final Conclusion

If I were reviewing this for a security readiness checkpoint, my conclusions would be:

Approved Concepts

* Signed immutable approval snapshot
* Delay window before broadcast
* Re-verification immediately before broadcast
* HMAC-SHA256
* Constant-time signature verification
* DB-level append-only audit records
* Fail-closed security_hold

Issues Requiring Attention Before Production Custody

1. Ensure destination includes all network-specific routing data (memo/tag/destination tag).
2. Ensure asset identity is signed if not already immutable.
3. Replace “never rotate secret” with key-versioned signatures.
4. Add explicit concurrency tests.
5. Document recovery procedures from security_hold.
6. Ensure verify and broadcast ownership transition cannot introduce a second TOCTOU window.

None of those findings invalidate the overall architecture. The architecture itself is fundamentally sound and follows the pattern I would expect for a custodial withdrawal approval pipeline.
